# SmartSpend v2 — Use Cases

> Все пользовательские сценарии продукта. Основа: PRODUCT_SPEC.md.

---

## Акторы

| Актор | Описание |
|---|---|
| **Гость** | Неаутентифицированный пользователь |
| **Пользователь** | Аутентифицированный пользователь с активным аккаунтом |
| **Автор** | Пользователь, создающий контент (статьи, наборы) |
| **Администратор** | Сотрудник SmartSpend, управляет официальными наборами |
| **Система** | Фоновые процессы: Scheduler, триггеры БД, SSE |

## Блок 1 — Аутентификация (UC-01…05)

---

### UC-01 | Регистрация по email

```
Актор:        Гость
Предусловие:  Пользователь не имеет аккаунта
Триггер:      Нажатие "Начать бесплатно" на лендинге
```

**Основной сценарий:**
1. Открывается AuthModal → вкладка "Регистрация"
2. Вводит имя, email, пароль
3. POST /api/auth/register → 201: `{ user, access_token }` + письмо с подтверждением
4. Редирект на `/onboarding` — квиз (3 вопроса о потреблении + имя)
5. POST /api/me/onboarding `{ answers, display_name }`
6. Редирект на `/feed`
7. Баннер: "Подтвердите email" (не блокирует доступ)

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Email уже зарегистрирован | Ошибка "Пользователь уже существует" |
| A2 | Пароль не соответствует требованиям | Инлайн-валидация до отправки |
| A3 | Сервер недоступен | Сообщение об ошибке, форма остаётся заполненной |

**Постусловие:**
- Аккаунт создан, статус `unverified`
- Созданы дефолтные группы инвентаря (Питание, Уход, Одежда, Быт, Здоровье, Разное)
- Access token выдан, refresh token установлен в httpOnly cookie

### UC-02 | Вход в аккаунт

```
Актор:        Гость
Предусловие:  Аккаунт существует
Триггер:      Нажатие "Войти" на лендинге или любой защищённой странице
```

**Основной сценарий:**
1. Открывается AuthModal → вкладка "Вход"
2. Вводит email + пароль
3. POST /api/auth/login → 200: `{ user, access_token }`
4. Редирект на `/feed`

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Неверный пароль | Ошибка "Неверный email или пароль" (не раскрывать что именно) |
| A2 | Аккаунт `suspended` | "Аккаунт временно заблокирован. Обратитесь в поддержку" |
| A3 | "Запомнить меня" включён | Refresh token живёт 30 дней вместо сессии |
| A4 | Access token истёк во время сессии | Система автоматически обновляет через POST /api/auth/refresh |

**Постусловие:** Пользователь аутентифицирован, получает доступ ко всем разделам

### UC-03 | Вход через OAuth (Яндекс / VK)

```
Актор:        Гость
Предусловие:  —
Триггер:      Нажатие "Войти через Яндекс" или "Войти через VK"
```

**Основной сценарий:**
1. Редирект на страницу авторизации OAuth-провайдера
2. Пользователь подтверждает доступ
3. Провайдер возвращает код → POST /api/auth/oauth/callback
4. Сервер находит или создаёт пользователя по email из провайдера
5. Если **новый пользователь** → редирект на `/onboarding` (квиз обязателен)
6. Если **существующий** → редирект на `/feed`

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Пользователь отклонил доступ в OAuth | Возврат на лендинг без авторизации |
| A2 | Email из OAuth уже зарегистрирован через пароль | Аккаунты связываются, вход выполнен |

**Постусловие:** Аккаунт создан или найден, `social_yandex_linked` / `social_vk_linked` = true

### UC-04 | Сброс пароля

```
Актор:        Гость
Предусловие:  Аккаунт с email существует
Триггер:      Нажатие "Забыли пароль?" в AuthModal
```

**Основной сценарий:**
1. Вводит email → POST /api/auth/forgot-password
2. Получает письмо со ссылкой (токен, TTL 1 час)
3. Переходит по ссылке → форма нового пароля
4. Вводит новый пароль → POST /api/auth/reset-password `{ token, password }`
5. Все refresh токены инвалидируются
6. Редирект на AuthModal с сообщением "Пароль изменён, войдите снова"

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Email не найден | Ответ всё равно "Письмо отправлено" (безопасность) |
| A2 | Токен истёк (> 1 ч) | "Ссылка устарела, запросите новую" |
| A3 | Токен уже использован | "Ссылка недействительна" |

**Постусловие:** Пароль изменён, все сессии завершены

### UC-05 | Удаление аккаунта

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован
Триггер:      Кнопка "Удалить аккаунт" в Settings → Account
```

**Основной сценарий:**
1. Показывается модал подтверждения с предупреждением
2. Пользователь вводит пароль для подтверждения
3. DELETE /api/me → статус аккаунта → `deleted`
4. Email и personal data анонимизируются немедленно
5. Все публикации остаются, но автор → `ghost` запись
6. Подписки, лайки, инвентарь, финансовые данные удаляются
7. Сессия завершается, редирект на лендинг

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Пользователь отменил | Модал закрывается, аккаунт цел |
| A2 | Неверный пароль | Ошибка "Неверный пароль" |

**Постусловие:**
- Аккаунт в статусе `deleted`, через 30 дней — автоматическая полная очистка (GDPR job)
- Публикации отображаются с автором "👻 Привидение"

### UC-06 | Редактирование профиля

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован, открыл Account
Триггер:      Нажатие "Редактировать" в разделе профиля

**Основной сценарий:**

**Альтернативные потоки:**

| A3 | Смена email | Требует подтверждения нового email → POST /api/auth/change-email |
| A4 | Смена пароля | Требует текущий пароль → POST /api/auth/change-password |

**Постусловие:** Профиль обновлён, аватар загружен на S3

### UC-10 | Добавление набора в профиль

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован, просматривает SetDetail
Триггер:      Нажатие "Добавить в профиль"
```

**Основной сценарий:**
1. POST /api/sets/:id/save
2. Тип добавленного набора определяет статус в профиле:
   - **Публичный набор** (`ss` или `community`) → добавляется в профиль пользователя как **публичный**
   - **Личный набор** → добавляется как **личный**
3. Сервер создаёт `envelope` для категории набора (если не существует)
4. Добавляет набор в `envelope_sets`
5. Для каждой позиции набора создаёт `inventory_item` со статусом `frozen`
6. Создаётся уведомление `inv_frozen`: "Добавлено N позиций в заморозку"
7. Редирект на `/inventory`, раздел "Заморозка"

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Набор уже добавлен | Ошибка или диалог "Добавить повторно?" |
| A2 | Пользователь редактирует параметры публичного набора после добавления | Набор автоматически переходит в статус **личного** — создаётся персональная копия, оригинал остаётся в каталоге |
| A3 | Публичный набор → личный (А2) | Видимость в каталоге пропадает, редактирование доступно |

**Постусловие:**
- Envelope создан/обновлён
- Все позиции набора в инвентаре со статусом `frozen`
- `set_saves` запись создана, счётчик `users_count` набора +1
- Если публичный — набор виден в профиле пользователя публично

### UC-10a | Удаление набора из профиля

```
Актор:        Пользователь
Предусловие:  Набор добавлен в профиль пользователя
Триггер:      Нажатие "Удалить из профиля" / "Скрыть" у набора
```

**Основной сценарий:**
1. DELETE /api/profile/sets/:id
2. Система проверяет статусы всех `inventory_item`, связанных с набором:
   - Позиции в статусе `frozen` (данные не заполнены) → **удаляются**
   - Позиции с заполненными данными (qty, price, purchase_date) → **переводятся в статус `paused`** и прикрепляются к набору **"Личное"** (персональный дефолтный набор пользователя, создаётся автоматически)
3. Набор удаляется из `envelope_sets` и из профиля пользователя
4. Для публичного набора — скрывается из каталога на странице пользователя

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | У набора нет позиций в инвентаре | Набор удаляется без дополнительных действий |
| A2 | Набор "Личное" не существует | Система создаёт его автоматически перед перекреплением |
| A3 | Все позиции набора в `frozen` (без данных) | Все удаляются, набор удаляется — нет paused позиций |

**Постусловие:**
- Набор удалён из профиля
- `frozen` позиции удалены
- Позиции с данными перешли в `paused`, прикреплены к набору "Личное"
- Счётчик `users_count` набора -1

### UC-10b | Перекрепление позиции инвентаря к другому набору

```
Актор:        Пользователь
Предусловие:  Позиция существует в инвентаре (статус `active`, `paused` или `frozen`)
Триггер:      Выбор "Прикрепить к набору" в меню позиции
```

**Основной сценарий:**
1. Открывается список доступных наборов пользователя (включая "Личное")
2. По умолчанию позиция прикреплена к набору **"Личное"** (если добавлена вручную или перенесена)
3. Пользователь выбирает новый набор
4. PUT /api/inventory/items/:id/set `{ set_id: <new_set_id> }`
5. Позиция открепляется от предыдущего набора и прикрепляется к новому
6. Группировка в инвентаре обновляется

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Выбранный набор — публичный (чужой) | Недоступен для прикрепления, только личные наборы пользователя |
| A2 | Позиция в статусе `frozen` | Перекрепление доступно, статус не меняется |
| A3 | Позиция уже прикреплена к выбранному набору | Без изменений, уведомление "Уже прикреплено" |

**Постусловие:**
- Позиция прикреплена к новому набору
- Инвентарь перегруппирован по набору

### UC-11 | Просмотр инвентаря

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован
Триггер:      Переход в раздел Инвентарь
```

**Основной сценарий:**
1. GET /api/inventory → группы с позициями

3. Отображаются кольца прогресса с цветовой кодировкой
4. Раздел "Заморозка" показывается если есть `frozen` позиции (серые, ❄️)

**Постусловие:** —

### UC-12 | Активация позиции из заморозки (frozen → active)

```
Актор:        Пользователь
Предусловие:  Позиция в статусе `frozen`
Триггер:      Нажатие "Купил" у позиции в заморозке
```

**Основной сценарий:**
1. Открывается форма: подтвердить/изменить qty, цену, дату покупки
2. PUT /api/inventory/items/:id/activate `{ purchase_date: today, qty, price }`
3. Статус → `active`, начинается расчёт сроков
4. Позиция перемещается из "Заморозки" в соответствующую группу
5. Если это была последняя frozen позиция набора → уведомление "Все позиции активированы ✅"

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Пользователь изменил qty | Расчёт идёт от нового значения |
| A2 | Пользователь указал прошедшую дату | Статус может сразу стать `soon` или `urgent` |

**Постусловие:** Позиция активна, участвует в расчёте финансового профиля

### UC-14 | Ручное добавление позиции в инвентарь

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован
Триггер:      Нажатие "Добавить вручную" в инвентаре
```

**Основной сценарий:**
1. Открывается форма: название, тип (consumable / wear), группа
2. Для consumable: qty, unit, daily_use, price, purchase_date
3. Для wear: wear_life_weeks, price, purchase_date
4. POST /api/inventory/items `{ ..., is_manual: true, set_id: 'personal' }`
5. Позиция автоматически прикрепляется к набору **"Личное"** (персональный дефолтный набор)
6. Позиция сразу в статусе `active` (не frozen, т.к. вручную)
7. Пользователь может переприкрепить позицию к другому набору через UC-10b

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Группа не существует | Создать новую группу или выбрать "Разное" |
| A2 | Набор "Личное" не создан | Система создаёт его автоматически |

**Постусловие:** Позиция создана с `is_manual=true`, прикреплена к набору "Личное", участвует в расчётах

### UC-16 | Система напоминает о заканчивающейся позиции

```
Актор:        Система (Scheduler)

**Основной сценарий:**

   - Создаётся notification `inv_reminder` (высокий приоритет): "Зубная паста закончилась"

### UC-35 | Уведомление о заморозке после добавления набора

```
Актор:        Система → Пользователь
Предусловие:  Пользователь добавил набор в конверт (UC-10)
Триггер:      Завершение POST /api/sets/:id/save
```

**Основной сценарий:**
1. Немедленно создаётся уведомление `inv_frozen`:
   "Добавлено 12 позиций в заморозку"
2. Пользователь видит уведомление в колоколе и тостер
3. Ежедневно (пока есть frozen позиции): напоминание "У вас N позиций в заморозке →"
4. При активации последней позиции: "Все позиции набора активированы ✅"

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Напоминание о заморозке уже отправлялось сегодня | Не дублировать, 1 раз в день |

**Постусловие:** Пользователь знает о замороженных позициях и мотивирован их активировать

## Блок 10a — Видимость и удаление контента (UC-38…39)

---

### UC-38 | Удаление статьи

```
Актор:        Автор
Предусловие:  Статья принадлежит автору
Триггер:      Нажатие "Удалить" в Account → Статьи
```

**Основной сценарий (публичная статья):**
1. Статья опубликована (`is_public=true`)
2. Кнопка "Редактировать" — недоступна
3. Кнопка "Удалить" — доступна
4. Показывается подтверждение: "Статья будет удалена. Это действие нельзя отменить."
5. Автор подтверждает → DELETE /api/articles/:id
6. Статья физически удаляется из системы
7. Исчезает из ленты, каталога, профиля автора
8. Комментарии к статье также удаляются

**Основной сценарий (личная статья):**
1. Статья личная (`is_public=false`)
2. Кнопки "Редактировать" и "Удалить" — обе доступны
3. Удаление → физически удаляется

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Автор отменил удаление | Модал закрывается, статья цела |
| A2 | Попытка редактировать публичную статью | Кнопка "Редактировать" скрыта / задизаблена; действие невозможно |

**Постусловие:** Статья удалена, `articles_count` автора −1, изображения удаляются из S3 (background job)

### UC-39 | Удаление личного набора

```
Актор:        Автор
Предусловие:  Набор является ЛИЧНЫМ (source=own), принадлежит автору
Триггер:      Нажатие "Удалить" в Account → Наборы
```

**Основной сценарий:**
1. Показывается подтверждение: "Личный набор будет удалён. Это действие нельзя отменить."
2. Автор подтверждает → DELETE /api/sets/:id
3. Набор физически удаляется
4. Исчезает из каталога (раздел "Мои") и Account
5. Если набор был добавлен автором в конверт — `envelope_set` запись удаляется, связанные `inventory_items` получают статус `archived`

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Набор публичный (source=community) | Кнопки "Удалить" нет — доступно только "Скрыть из каталога" (→ UC-28a) |
| A2 | Автор отменил | Модал закрывается, набор цел |

**Постусловие:** Личный набор удалён из БД; связанные позиции инвентаря архивированы

### UC-36 | Управление аккаунтом удалённого пользователя (Ghost)

```
Актор:        Система
Предусловие:  Пользователь выполнил UC-05 (удалил аккаунт)
Триггер:      Статус аккаунта изменился на `deleted`
```

**Основной сценарий:**
1. Немедленно: email и personal data анонимизируются
2. Все публикации (статьи, наборы) — `author_id` → ссылка на `ghost_user` (системный пользователь)
3. В ленте: "👻 Привидение" вместо имени автора
4. Клик по автору → GhostProfile: "Этот аккаунт был удалён. Контент остаётся доступным"
5. Через 30 дней: фоновый job выполняет полную очистку данных пользователя

**Постусловие:** Пользователь полностью удалён через 30 дней, контент сохранён

### UC-37 | Поиск по контенту

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован (опционально для MVP)
Триггер:      Ввод запроса в поисковую строку
```

**Основной сценарий:**
1. GET /api/search?q=&type=sets|articles|users
2. Система ищет по:
   - Наборы: title, desc, items.name
   - Статьи: title, excerpt, body (первые 500 символов)
   - Пользователи: display_name, handle, bio
3. Результаты сгруппированы по типу
4. Клик на результат → соответствующая страница

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Запрос < 2 символов | Поиск не выполняется |
| A2 | Нет результатов | "Ничего не найдено по запросу «...»" |

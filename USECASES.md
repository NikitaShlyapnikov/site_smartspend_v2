## Акторы

| Актор | Описание |
|---|---|
| **Гость** | Неаутентифицированный пользователь |
| **Пользователь** | Аутентифицированный пользователь с активным аккаунтом |
| **Автор** | Пользователь, создающий контент (статьи, наборы) |
| **Администратор** | Сотрудник SmartSpend, управляет официальными наборами |
| **Система** | Фоновые процессы: Scheduler, триггеры БД, SSE |

## Блок 0 — Гостевой режим (UC-01)

---

### UC-01 | Просмотр контента без регистрации (гостевой режим)

```
Актор:        Гость
Предусловие:  Пользователь не авторизован
Триггер:      Переход на публичную страницу (набор, статья)
```

**Публичные страницы (доступны гостю):**
- `/set/:id` — детальная страница набора
- `/article/:id` — чтение статьи

**Гостевой интерфейс:**
1. Гость может просматривать контент в полном объёме
2. При попытке совершить требующее авторизации действие — открывается AuthModal

**Защищённые страницы — показывают AuthModal поверх:**

| Страница | Поведение при попытке гостя открыть |
|---|---|
| `/feed` | AuthModal, модал нельзя закрыть без входа |
| `/profile` | AuthModal, модал нельзя закрыть без входа |
| `/account` | AuthModal, модал нельзя закрыть без входа |
| `/settings` | AuthModal, модал нельзя закрыть без входа |
| `/inventory` | AuthModal, модал нельзя закрыть без входа |
| `/notifications` | AuthModal, модал нельзя закрыть без входа |
| `/create-set` | AuthModal, модал нельзя закрыть без входа |
| `/create-article` | AuthModal, модал нельзя закрыть без входа |

**Постусловие после входа из AuthModal:**
- Пользователь остаётся на той же странице (не редиректится на /profile)
- Страница перезагружается и теперь отображается с полным сайдбаром

### UC-02 | Регистрация по email

```
Актор:        Гость
Предусловие:  Пользователь не имеет аккаунта
Триггер:      Нажатие "Создать аккаунт" на лендинге
```

**Основной сценарий:**
1. Открывается AuthModal → вкладка "Регистрация"
2. Вводит имя, email, пароль
3. POST /api/auth/register → 201: `{ user, access_token }` + письмо с подтверждением
4. Редирект на `/onboarding` — квиз (3 вопроса о потреблении + имя)
5. POST /api/me/onboarding `{ answers, display_name }`
6. Редирект на `/profile`
7. Баннер: "Подтвердите email" (не блокирует доступ)

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Email уже зарегистрирован | Ошибка "Пользователь уже существует" |
| A2 | Пароль не соответствует требованиям | Инлайн-валидация до отправки |
| A3 | Сервер недоступен | Сообщение об ошибке, форма остаётся заполненной |

**Постусловие:**
- Аккаунт создан, статус `unverified`
- Созданы дефолтные группы инвентаря (Питание, Уход, Одежда, Быт, Здоровье, Разное)
- Access token выдан, refresh token установлен в httpOnly cookie

### UC-03 | Вход в аккаунт

```
Актор:        Гость
Предусловие:  Аккаунт существует
Триггер:      Нажатие "Войти" на лендинге или любой защищённой странице
```

**Основной сценарий:**

4. Редирект на `/profile`

**Альтернативные потоки:**

|---|---|---|
| A1 | Неверный пароль | Ошибка "Неверный email или пароль" (не раскрывать что именно) |

| A4 | Access token истёк во время сессии | Система автоматически обновляет через POST /api/auth/refresh |

**Постусловие:** Пользователь аутентифицирован, получает доступ ко всем разделам

```
Актор:        Гость
Предусловие:  —
Триггер:      Нажатие "Войти через Яндекс" или "Войти через VK"
```

### UC-05 | Сброс пароля

```
Актор:        Гость
Предусловие:  Аккаунт с email существует
Триггер:      Нажатие "Забыли пароль?" в AuthModal
```

**Основной сценарий:**
1. Вводит email → POST /api/auth/forgot-password
2. Получает письмо со ссылкой (токен, TTL 1 час)
3. Переходит по ссылке → форма нового пароля
4. Вводит новый пароль → POST /api/auth/reset-password `{ token, password }`
5. Все refresh токены инвалидируются
6. Редирект на AuthModal с сообщением "Пароль изменён, войдите снова"

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Email не найден | Ответ всё равно "Письмо отправлено" (безопасность) |
| A2 | Токен истёк (> 1 ч) | "Ссылка устарела, запросите новую" |
| A3 | Токен уже использован | "Ссылка недействительна" |

**Постусловие:** Пароль изменён, все сессии завершены

### UC-06 | Удаление аккаунта

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован
Триггер:      Кнопка "Удалить аккаунт" в Settings → Account
```

**Основной сценарий:**
1. Показывается модал подтверждения с предупреждением
2. Пользователь вводит пароль для подтверждения
3. DELETE /api/me → запускается таймер удаления (7 дней)
4. Email и personal data анонимизируются немедленно
5. Сессия завершается, редирект на лендинг
6. Ежедневно в течение 7 дней пользователь получает уведомление: "Аккаунт будет удалён через N дней — [Отменить удаление]"
7. По истечении 7 дней — аккаунт и все связанные данные удаляются окончательно

**Что удаляется:**
- Лайки и комментарии пользователя на чужих материалах
- Инвентарь и все позиции
- Финансовые данные (доходы, конверты, SmartBase)
- Подписки (входящие и исходящие)
- Уведомления
- Личные наборы (`source=own`)

**Что остаётся (под именем "👻 Привидение"):**
- Публичные наборы (`source=community`) — остаются в каталоге; другие пользователи продолжают ими пользоваться
- Публичные статьи — остаются в ленте; клик на автора → "Этот аккаунт был удалён"

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Пользователь отменил из уведомления в период 7 дней | PUT /api/me/cancel-deletion → аккаунт восстанавливается, таймер сбрасывается |
| A2 | Пользователь отменил через модал | Модал закрывается, аккаунт цел |
| A3 | Неверный пароль | Ошибка "Неверный пароль" |
| A4 | У пользователя есть публичные наборы | Модал дополнительно предупреждает: "Ваши публичные наборы останутся в каталоге под анонимным именем" |

**Постусловие:**
- Аккаунт в статусе `pending_deletion`, через 7 дней — полное удаление
- Публичные наборы и статьи сохранены с автором "👻 Привидение"

### UC-08 | Редактирование профиля

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован, открыл Account
Триггер:      Нажатие "Редактировать" в разделе профиля
```

**Основной сценарий:**

**Альтернативные потоки:**

| A3 | Смена email | Требует подтверждения нового email → POST /api/auth/change-email |
| A4 | Смена пароля | Требует текущий пароль → POST /api/auth/change-password |

**Постусловие:** Профиль обновлён, аватар загружен на S3

### UC-11 | Изменение настроек

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован
Триггер:      Переход в Settings (/settings)
```

### UC-12 | Заполнение финансовых данных

```
Актор:        Пользователь
Предусловие:  Пользователь авторизован
Триггер:      Нажатие кнопки "Редактировать" в финансовом профиле
```

**Основной сценарий:**
1. Открывается квиз-опросник (FinancialQuiz)
2. Пользователь последовательно отвечает на вопросы:
   - Ежемесячный доход
   - Фиксированные расходы: жильё, кредит, еда, транспорт, прочее
   - Текущий капитал
   - Ставка EmoSpend (выбор режима)
3. PUT /api/profile/finance
4. Система моментально считает и показывает:
   - Smart-базу (расходы на все наборы в конвертах)
   - Свободный остаток
   - Отчисление в ядро
   - EmoSpend
   - График прогноза капитала на 10 лет

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Нет активных наборов в конвертах | Smart-база = 0, расчёт всё равно работает |
| A2 | Фиксированные расходы > дохода | Свободный остаток < 0, предупреждение |

**Постусловие:** `user_finance` заполнен, расчёты доступны

### UC-14 | Автоматический пересчёт SmartBase

```
Актор:        Система
Предусловие:  У пользователя есть активные наборы в конвертах
Триггер:      Пользователь добавил или удалил набор в конверте
```

**Основной сценарий:**
1. После изменения состава конверта сервер пересчитывает:
   - Суммарные расходы **по конкретному конверту** (сумма расходов всех наборов в нём)
   - **Общую** Smart-базу (сумма по всем конвертам)
   ```
   consumable: price * (daily_use * 30) / qty
   wear:       price / (wear_life_weeks * 7 / 30)
   ```
2. Возвращается обновлённая сводка GET /api/profile/summary
3. Frontend обновляет цифры в Profile без перезагрузки

**Постусловие:** Расходы по конверту и общая Smart-база актуальны

### UC-15 | Просмотр и фильтрация каталога

```
Актор:        Пользователь (Гость — только просмотр)
Предусловие:  —
Триггер:      Переход в раздел Каталог
```

**Основной сценарий:**
1. GET /api/catalog → список **публичных** наборов с пагинацией (личные наборы `source=own` в каталоге не отображаются — доступны только владельцу через фильтр «Мои»)
2. Пользователь применяет фильтры: категория, источник (SS / Community / Мои), тип (база / дополнение)
3. Меняет сортировку: новые / популярные за день/неделю/всё время
4. Использует поиск по составу: вводит название позиции (например, «рис») → выводятся наборы, содержащие эту позицию; совпадающие позиции подсвечиваются в карточке
5. Карточка набора: название, описание, категория, позиции (с подсветкой совпадений), лайки, сколько используют

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Нет наборов по фильтру | Пустое состояние "Наборов не найдено" |
| A2 | Подгрузка следующей страницы | Infinite scroll / кнопка "Загрузить ещё" |
| A3 | Поиск по составу + фильтр по категории | Оба условия применяются одновременно |
| A4 | Очистка строки поиска | Отображаются все наборы по текущим фильтрам |
| A5 | Гость нажимает «Понравившиеся» | Открывается AuthModal (требуется авторизация) |
| A6 | Гость нажимает «Мои» | Открывается AuthModal (требуется авторизация) |

**Постусловие:** —

### UC-17 | Добавление набора в профиль

```
Актор:        Пользователь
```

### UC-32 | Комментирование статьи

```
Актор:        Пользователь
Предусловие:  Статья опубликована, пользователь авторизован
Триггер:      Ввод текста в поле комментария на странице статьи
```

**Основной сценарий:**
1. Пользователь вводит текст → POST /api/articles/:id/comments
2. Комментарий появляется в списке на странице статьи
3. Автор статьи получает уведомление `comment_on_my`
4. Другой пользователь нажимает "Ответить" под комментарием → в поле ввода автоматически подставляется `@никнейм` адресата; отправка создаёт ответ с `parent_id` → автор оригинального комментария получает `comment_reply`
5. Лайк комментария → POST /api/comments/:id/like
6. Если комментарий принадлежит текущему пользователю — рядом отображается кнопка "Удалить"

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Удаление своего комментария | DELETE /api/comments/:id → `is_deleted=true`, текст → "[удалено]" |

**Примечание:** Комментарии пользователя НЕ отображаются отдельной вкладкой в Account. Просмотр всех комментариев возможен только в контексте конкретной статьи.

**Постусловие:** `comments_count` статьи +1

### UC-33 | Удаление статьи

```
Актор:        Автор
Предусловие:  Статья принадлежит автору
Триггер:      Нажатие "Удалить" в Account → Статьи
```

**Основной сценарий (публичная статья):**
1. Статья опубликована (`is_public=true`)
2. Кнопка "Редактировать" — недоступна
3. Кнопка "Удалить" — доступна
4. Показывается подтверждение: "Статья будет удалена. Это действие нельзя отменить."
5. Автор подтверждает → DELETE /api/articles/:id
6. Статья физически удаляется из системы
7. Исчезает из ленты, каталога, профиля автора
8. Комментарии к статье также удаляются

**Основной сценарий (личная статья):**
1. Статья личная (`is_public=false`)
2. Кнопки "Редактировать" и "Удалить" — обе доступны
3. Удаление → физически удаляется

**Альтернативные потоки:**

| ID | Условие | Результат |
|---|---|---|
| A1 | Автор отменил удаление | Модал закрывается, статья цела |
| A2 | Попытка редактировать публичную статью | Кнопка "Редактировать" скрыта / задизаблена; действие невозможно |

**Постусловие:** Статья удалена, `articles_count` автора −1, изображения удаляются из S3 (background job)

